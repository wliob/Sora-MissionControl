# Admin Adapter Contract - Phase 6

This document outlines the administrative adapter contract for Phase 6 of Sora-MissionControl, focusing on interaction with Hermes CLI/API for cron jobs, webhooks, skills, and MCP. It maps observed commands and outputs to proposed TypeScript interfaces, identifies unavailable routes, and recommends a browser-safe boundary.

## 1. Real Commands/Routes Tested and Sample Outputs

### 1.1 Cron Jobs

**Commands:**
*   `hermes cron list`
*   `default_api.cronjob(action="list")`

**Sample Output (empty list):**
```json
{
  "cronjob_response": {
    "count": 0,
    "jobs": [],
    "success": true
  }
}
```

**Mapping to TS Interface (CronJob):**

```typescript
interface CronJob {
  id: string; // Auto-generated by Hermes
  name: string; // Optional, human-friendly name
  schedule: string; // e.g., "30m", "every 2h", "0 9 * * *", ISO timestamp
  prompt: string; // Self-contained prompt for agent
  skills?: string[]; // List of skill names to load
  enabledToolsets?: string[]; // List of toolset names
  model?: { // Optional model override
    model: string;
    provider?: string;
  };
  script?: string; // Path to a script
  noAgent?: boolean; // If true, script runs without LLM
  deliver?: string; // Delivery target
  repeat?: number; // Repeat count
  contextFrom?: string[]; // Job IDs for context injection
  workdir?: string; // Working directory
  // Additional fields from 'hermes cron list' output (if available) would be added here
}

interface CronJobStatus {
  success: boolean;
  count: number;
  jobs: CronJob[];
}
```

The `hermes secrets` command indicates integration with external secret managers (e.g., Bitwarden). Direct programmatic access to list/create/delete individual secrets via a CLI/API appears UNAVAILABLE. The interaction is primarily around configuring the *integration* with a secret manager.

## 2. Unavailable Routes & Blockers

*   **Secrets Management (Direct):** Direct listing, creation, or deletion of individual secrets via Hermes CLI/API is UNAVAILABLE. The current mechanism relies on integration with external secret managers (e.g., Bitwarden).
    *   **Blocker:** No explicit Hermes API endpoints for granular secret management. The tool focuses on *integrating* with external systems, not being a secret store itself.
    *   **Safe UI State:** The UI should reflect the status of the *integration* with the secret manager (e.g., "Bitwarden connected/disconnected") and provide guidance on how to manage secrets within the external system. No attempt should be made to display or edit individual secrets.

## 3. Recommendation for Browser-Safe Boundary

To ensure a browser-safe boundary for Sora-MissionControl's admin adapter, all interactions with the Hermes CLI/API should be proxied through a local server endpoint, rather than attempting direct client-side execution.

**Recommended Proxy Routes (Examples):**

*   `/api/admin/cron/list` -> `hermes cron list`
*   `/api/admin/cron/create` -> `hermes cron create ...`
*   `/api/admin/webhooks/list` -> `hermes webhook list`
*   `/api/admin/webhooks/add` -> `hermes webhook add ...`
*   `/api/admin/skills/list` -> `hermes skills list`
*   `/api/admin/skills/view/:name` -> `hermes skills view :name`
*   `/api/admin/mcp/list` -> `hermes mcp list`
*   `/api/admin/mcp/test/:name` -> `hermes mcp test :name`
*   `/api/admin/secrets/status` -> Internal check of secret manager integration status (not direct `hermes secrets` call, which is for configuration)

This approach separates the frontend from direct shell execution, allows for proper authentication/authorization at the server level, and prevents direct exposure of sensitive CLI commands or environment variables to the browser. API keys and other credentials used to execute Hermes commands on the backend would reside *only* on the server.
